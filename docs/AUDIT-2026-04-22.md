# Automated Pipeline Audit — 2026-04-22

**Question:** When run on GitHub Actions (no human in the loop), does our pipeline correctly discover new positions as they appear?

**Short answer:** Mostly yes, with **4 gaps** that cause false negatives.

---

## The real architecture (how it works today)

```
Daily 07:00 UTC (recon-daily.yml):
  └── src/build-wallet-inventory.js
  └── src/build-debank-recon.js
      └── DeBank call per wallet → data/recon/debank-wallet-summary.json
          (wallet, chain, total_usd, protocols[])
      └── filter: chains with total_usd ≥ $50K
  └── src/reconcile-gaps.js → gap-report.json
  └── commit

Hourly :15 (free-scans-hourly.yml):
  └── src/build-alchemy-recon.js  ← Alchemy token balances for every active wallet+chain
  └── src/build-canonical-token-matches.js  ← match addresses vs registries
  └── src/token-discovery.js      ← Layer 2: vault / YBS / wallet-held
  └── src/aave-scanner.js         ← Aave GraphQL, gated by hasProtocolHint(aave3)
  └── src/morpho-scanner.js       ← Morpho REST, gated by hasProtocolHint(morpho)
  └── src/euler-scanner.js        ← subgraph + RPC, NO protocol hint gate
  └── src/spark-scanner.js        ← on-chain reads, NO protocol hint gate
  └── src/fluid-scanner.js        ← REST + RPC, NO protocol hint gate
  └── src/pendle-scanner.js       ← PT/YT scan, NO protocol hint gate
  └── src/enrich-markets.js
  └── src/fetch-base-apy.js
  └── src/fetch-merkl.js
  └── src/export.js → data.json
  └── src/source-audit.js
  └── src/validate.js
  └── commit
```

## What works correctly

1. **DeBank is used ONLY for recon** (wallet+chain value map + protocol hints). Scanners never insert a DeBank row into the position lane.

2. **Daily recon auto-updates** `debank-wallet-summary.json`. If a wallet moves money to a new chain, the daily run picks it up within 24h.

3. **Token discovery (Layer 2)** scans every active wallet+chain and correctly classifies tokens as vault / YBS / wallet-held, with $50K threshold.

4. **Most scanners are DeBank-independent.** Euler, Spark, Fluid, Pendle all scan any active wallet (don't require DeBank to flag the protocol).

5. **Hourly refresh** means position changes propagate within ~60 minutes.

## The 4 gaps

### GAP 1: Aave + Morpho require DeBank protocol hints

**Where:** `src/aave-scanner.js` line 242 and `src/morpho-scanner.js` line 276:

```javascript
if (!hasProtocolHint(row, ['aave3', 'aave-v3', ...])) continue;
if (!hasProtocolHint(row, ['morphoblue', 'morpho'])) continue;
```

**Impact:** If DeBank doesn't mention Aave/Morpho in its recon for a wallet+chain, the scanner skips entirely. This is the thing that bit us this morning — I ran aave-scanner manually to find the $109M Plasma position. But DeBank *did* flag plasma_aave3 for that wallet, so the hourly would have caught it. The vulnerability is:
- DeBank rate-limited during daily recon → misses a chain
- Wallet opens Aave position on a new obscure chain → DeBank doesn't yet know
- DeBank changes their protocol naming convention → our hint list stops matching

**Fix:** Remove the `hasProtocolHint` gate. Scan every wallet+chain ≥ $50K. The scanner APIs return empty for wallets with no position anyway, so there's no cost penalty.

### GAP 2: Chain must be ≥ $50K in DeBank recon total

**Where:** `recon-helpers.js` line 4, `build-debank-recon.js` threshold.

**Impact:** If DeBank sees $49K on a chain, we never scan it. This is mostly correct (threshold for focus) but can miss positions with equal supply+borrow (net ≈ $0).

Example: Wallet supplies $10M USDe + borrows $10M USDT → net $0 → DeBank chain total ≈ $0 → we never scan Aave for that chain.

**Fix:** Lower threshold to ~$10K, or use a "known positions" rule (always scan ETH/base/arb/mnt/plasma for whales regardless of DeBank total).

### GAP 3: Scanner hasn't been re-run since DeBank refresh

**Where:** Daily recon runs at 07:00, hourly scans run at :15 every hour. So between 07:00 and 07:15, there's a 15-min window where DeBank data is fresh but scanners haven't re-read it. If a wallet acquires new Aave position at 07:00, the 07:15 scan picks it up. So this window is fine.

**Real issue:** If the hourly workflow fails (rate limits, network, bug), positions go stale. Example: aave-scanner didn't run yesterday → Plasma $109M invisible until I manually ran it this morning.

**Fix:**
- Each scanner must be idempotent (they are).
- Hourly workflow needs retries on transient failures.
- `validate.js` should warn if any scanner hasn't written in > 2 hours.

### GAP 4: New chains / new protocols not in config

**Where:** Each scanner has a hardcoded chain list.

**Impact:** New Aave deployment on HyperLiquid → scanner doesn't know, positions missed.

**Fix:** This is manual — we update the chain list when Aave deploys new chain. Should be documented.

## What DeBank does vs what our scanners do

| Data | Source | Notes |
|---|---|---|
| Wallet list | `data/whales.json` (manual) | We control this |
| Chains per wallet | DeBank recon | Refreshed daily |
| USD value per chain | DeBank recon | Used for $50K gate, not for position data |
| Protocol hints | DeBank recon | Aave/Morpho gate — GAP 1 |
| Actual position data | Scanner APIs + RPC | Authoritative |
| Token identities | Local registry (CoinGecko mirror) | Authoritative |
| YBS APYs | `data/stables.json` | Authoritative |
| Vault APYs | `data/vaults.json` | Authoritative |
| DeBank position rows in DB? | None (purged 2026-04-22) | ✓ clean |

**DeBank is used for:**
1. Discovering which (wallet, chain) pairs exist — necessary
2. Prioritizing which chains to scan — optimization
3. Gap report (comparing our total vs DeBank total) — sanity check

**DeBank is NOT used for:**
- Writing positions to DB
- Computing APYs
- Pricing tokens
- Token identities

## The test: "New whale added, no human intervention"

If we add a new whale address to `data/whales.json` with positions on Aave + Euler + a random token holding, what happens?

**Day 1 morning:**
- `recon-daily.yml` runs 07:00.
- `build-wallet-inventory.js` picks up the new wallet.
- `build-debank-recon.js` fetches DeBank data → adds rows to summary.
- Wallet appears in `debank-wallet-summary.json` with chain totals and protocol hints.

**Day 1 at :15 of same hour (or next hour):**
- `free-scans-hourly.yml` runs.
- `build-alchemy-recon.js` scans new wallet's tokens.
- `token-discovery.js` classifies held tokens (vault / YBS / wallet-held).
- `aave-scanner.js` scans only if DeBank flagged Aave hint. ⚠️ **GAP 1**
- `euler-scanner.js` scans all active wallets. ✓
- Export runs. New positions appear in `data.json`.

**Conclusion:** Yes, we auto-discover — with caveats around GAP 1 (Aave/Morpho need DeBank to mention the protocol) and GAP 2 (chain total < $50K is invisible).

If we fix GAP 1 (remove hint gates), the pipeline fully auto-discovers new positions on Day 1.
